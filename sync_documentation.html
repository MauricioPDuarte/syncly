🔄 Visão Geral do Sistema

📋 O que é o Syncly?

O Syncly é um sistema avançado de sincronização para Flutter que mantém os dados do aplicativo móvel sincronizados com o servidor backend. Utiliza uma arquitetura baseada em estratégias com configuração centralizada via SyncConfig, oferecendo flexibilidade máxima e facilidade de uso.

✨ Principais Funcionalidades

  • • Arquitetura baseada em estratégias
  • • Sincronização incremental - baixa apenas dados modificados
  • • Configuração centralizada via SyncConfig
  • • Injeção de dependência com GetIt
  • • Sistema de notificações integrado
  • • Modo offline com fila de operações
  • • Upload de arquivos e autenticação
  • • Retry automático e logging detalhado

🎯 Objetivo Principal

Garantir que todas as operações realizadas pelo usuário (criar, editar, deletar) sejam sincronizadas com o servidor, e que os dados mais recentes sejam baixados automaticamente, mesmo em condições de conectividade instável.

🚀 Sincronização Incremental

🎯 Nova Funcionalidade

A sincronização incremental é uma otimização que permite baixar apenas dados novos, modificados ou excluídos, em vez de apagar todos os dados locais e baixar tudo novamente a cada sincronização.

✨ Benefícios da Sincronização Incremental

⚡ Performance

Sincronizações até 90% mais rápidas para grandes volumes de dados, baixando apenas o que realmente mudou.

📱 Economia de Dados

Redução significativa no uso de dados móveis, especialmente importante para usuários com planos limitados.

🔋 Menor Consumo

Reduz o uso de bateria e processamento, melhorando a experiência geral do usuário.

🛡️ Segurança

Fallback automático para sincronização completa quando necessário, garantindo consistência dos dados.

🔄 Como Funciona

1

Verificação de Timestamp

O sistema verifica o timestamp da última sincronização bem-sucedida armazenado localmente.

2

Decisão Inteligente

Determina se deve usar sincronização incremental ou completa baseado na configuração e intervalo de tempo.

3

Requisição ao Backend

Envia o timestamp para o backend, que retorna apenas dados modificados desde essa data.

4

Processamento Seletivo

Aplica apenas as mudanças necessárias: novos dados, atualizações e remoções específicas.

5

Atualização do Timestamp

Salva o novo timestamp da sincronização para uso na próxima execução.

⚙️ Configuração no Flutter

class MeuSyncConfig extends SyncConfig {
  @override
  bool get useIncrementalSync => true;
  
  @override
  Duration get maxIncrementalSyncInterval => const Duration(days: 7);
  
  @override
  Future<DateTime?> getLastSyncTimestamp() async {
    // Implementar persistência do timestamp
    final prefs = await SharedPreferences.getInstance();
    final timestamp = prefs.getString('last_sync_timestamp');
    return timestamp != null ? DateTime.parse(timestamp) : null;
  }
  
  @override
  Future<void> saveLastSyncTimestamp(DateTime timestamp) async {
    // Salvar timestamp da última sincronização
    final prefs = await SharedPreferences.getInstance();
    await prefs.setString('last_sync_timestamp', timestamp.toIso8601String());
  }
  

}

💡 Dica Importante

O maxIncrementalSyncInterval define o tempo máximo para usar sincronização incremental. Após esse período, o sistema automaticamente fará uma sincronização completa para garantir consistência.

🔧 Guia de Implementação do Backend

📋 Para Desenvolvedores Backend

Esta seção explica como implementar os endpoints do backend para suportar a sincronização incremental do Syncly. É essencial seguir este formato para garantir compatibilidade total.

📤 Formato da Requisição do Cliente

O Syncly enviará requisições com o seguinte formato:

GET /api/sync/data

Headers:
Authorization: Bearer <token>
Content-Type: application/json

Query Parameters:
last_sync: 2024-01-15T10:30:00.000Z  (opcional - para sincronização incremental)
user_id: 12345                        (se necessário para filtrar dados do usuário)

🔍 Lógica de Decisão

  • • Se last_sync está presente: Retornar apenas dados modificados desde essa data
  • • Se last_sync está ausente: Retornar todos os dados (sincronização completa)

O backend deve retornar dados no seguinte formato JSON:

✅ success

Indica se a operação foi bem-sucedida

🕐 sync_timestamp

Timestamp atual do servidor (ISO 8601)

🔄 is_incremental

Indica se foi sincronização incremental

📊 Estrutura dos Dados

🆕 created

Array com registros criados desde o last_sync

  • • Deve incluir todos os campos do registro
  • • Obrigatório: id, created_at, updated_at
  • • Usar formato ISO 8601 para datas

🔄 updated

Array com registros modificados desde o last_sync

  • • Deve incluir todos os campos atualizados
  • • O updated_at deve ser posterior ao last_sync
  • • Incluir registro completo, não apenas campos alterados

🗑️ deleted

Array com IDs dos registros excluídos desde o last_sync

  • • Apenas os IDs dos registros excluídos
  • • O cliente removerá estes registros localmente
  • • Importante para manter consistência dos dados

💾 Exemplo de Implementação SQL

Estrutura de Tabelas Recomendada:

-- Adicionar colunas de auditoria em todas as tabelas
ALTER TABLE users ADD COLUMN created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP;
ALTER TABLE users ADD COLUMN updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP;
ALTER TABLE users ADD COLUMN deleted_at TIMESTAMP NULL;

-- Criar índices para performance
CREATE INDEX idx_users_updated_at ON users(updated_at);
CREATE INDEX idx_users_deleted_at ON users(deleted_at);

Query para Sincronização Incremental:

-- Registros criados
SELECT * FROM users 
WHERE created_at > ? AND deleted_at IS NULL
ORDER BY created_at;

-- Registros atualizados (mas não criados no período)
SELECT * FROM users 
WHERE updated_at > ? AND created_at <= ? AND deleted_at IS NULL
ORDER BY updated_at;

-- Registros excluídos (soft delete)
SELECT id FROM users 
WHERE deleted_at > ?
ORDER BY deleted_at;

⚠️ Considerações Importantes

  • • Use soft delete (deleted_at) em vez de DELETE físico
  • • Mantenha registros excluídos por pelo menos 30 dias
  • • Use transações para garantir consistência
  • • Considere paginação para grandes volumes de dados

🚀 Exemplo de Implementação (Node.js)

app.get('/api/sync/data', async (req, res) => {
  try {
    const { last_sync, user_id } = req.query;
    const currentTimestamp = new Date().toISOString();
    const isIncremental = !!last_sync;
    
    let result = {
      success: true,
      sync_timestamp: currentTimestamp,
      is_incremental: isIncremental,
      data: {}
    };
    
    if (isIncremental) {
      // Sincronização incremental
      const syncDate = new Date(last_sync);
      
      // Buscar dados de usuários
      const usersCreated = await db.query(
        'SELECT * FROM users WHERE created_at > ? AND deleted_at IS NULL',
        [syncDate]
      );
      
      const usersUpdated = await db.query(
        'SELECT * FROM users WHERE updated_at > ? AND created_at <= ? AND deleted_at IS NULL',
        [syncDate, syncDate]
      );
      
      const usersDeleted = await db.query(
        'SELECT id FROM users WHERE deleted_at > ?',
        [syncDate]
      );
      
      result.data.users = {
        created: usersCreated,
        updated: usersUpdated,
        deleted: usersDeleted.map(u => u.id)
      };
      
      // Repetir para outras entidades (products, orders, etc.)
      
    } else {
      // Sincronização completa
      const allUsers = await db.query(
        'SELECT * FROM users WHERE deleted_at IS NULL'
      );
      
      result.data.users = {
        created: allUsers,
        updated: [],
        deleted: []
      };
    }
    
    res.json(result);
    
  } catch (error) {
    console.error('Erro na sincronização:', error);
    res.status(500).json({
      success: false,
      error: 'Erro interno do servidor'
    });
  }
});

⚙️ Componentes Principais

🎯 SyncService

Implementação principal do ISyncService. Gerencia todo o ciclo de vida da sincronização, incluindo operações automáticas, tratamento de falhas e modo offline.

  • • Controle de mutex para operações concorrentes
  • • Timer para sincronização automática
  • • Gerenciamento de conectividade
  • • Recovery automático de falhas

📱 BackgroundSyncService

Gerencia sincronização em background usando WorkManager. Executa sync mesmo quando o app está fechado.

  • • Integração com WorkManager
  • • Notificações de progresso
  • • Execução independente do app
  • • Configuração de intervalos

🔄 Estratégias de Sync

SyncDownloadStrategy e SyncUploadStrategy implementam a lógica específica de download e upload.

  • • Download: limpeza, cache, notificações
  • • Upload: fila, retry, batch processing
  • • Estratégias plugáveis via IDownloadStrategy
  • • Orquestração de múltiplas estratégias

📝 Sistema de Logs

LoggerProvider e InternalSyncLogStorage gerenciam logs de sincronização de forma independente.

  • • Storage independente via IStorageProvider
  • • Logs estruturados com metadata
  • • Retry automático e cleanup
  • • Debug logging separado

🔄 Fluxo de Dados Interno

1. Operação de Log (Create/Update/Delete)

↓

↓

2. Ciclo de Sincronização

↓

↓

↓

SyncService._updateSyncStatus() → syncData.value = SyncData(...)

↓

↓

Controllers/Widgets reagem às mudanças de estado

🌐 Rotas da API de Sincronização

📋 Visão Geral

🌐 1.1. Implementar Métodos HTTP

Implemente os métodos HTTP usando Dio ou outro cliente HTTP:

  // ========== MÉTODOS HTTP ==========
  @override
  Future> httpGet(
    String url, {
    Map? headers,
    Map? queryParameters,
    Duration? timeout,
  }) async {
    try {
      final response = await _dio.get(
        url,
        queryParameters: queryParameters,
        options: Options(
          headers: {...defaultHeaders, ...?headers},
          receiveTimeout: timeout ?? networkTimeout,
        ),
      );
      
      return SyncHttpResponse(
        data: response.data,
        statusCode: response.statusCode ?? 0,
        headers: response.headers.map,
        isSuccess: response.statusCode != null && response.statusCode! >= 200 && response.statusCode! < 300,
      );
    } catch (e) {
      return SyncHttpResponse(
        data: null,
        statusCode: 0,
        headers: {},
        isSuccess: false,
        error: e.toString(),
      );
    }
  }
  
  @override
  Future> httpPost(
    String url, {
    dynamic data,
    Map? headers,
    Map? queryParameters,
    Duration? timeout,
  }) async {
    try {
      final response = await _dio.post(
        url,
        data: data,
        queryParameters: queryParameters,
        options: Options(
          headers: {...defaultHeaders, ...?headers},

        ),

        data: response.data,

        headers: response.headers.map,

      return SyncHttpResponse(
        data: null,
        statusCode: 0,
        headers: {},

🎮 4. Monitoramento nos Controllers

Monitore o estado de sincronização nos controllers:

class HomeController extends BaseController {
  AppSyncService get syncService => SyncConfigurator.syncService;

  @override
  void onInit() {
    super.onInit();
    // Escuta mudanças no estado de sincronização
    syncService.syncData.addListener(_onSyncStateChanged);
  }

  void _onSyncStateChanged() {
    final syncData = syncService.syncData.value;
    
    if (syncData.status == AppSyncStatus.success || 
        syncData.status == AppSyncStatus.idle) {
      // Recarregar dados quando sincronização terminar
      _loadData();
    }
  }
  
  void forceSync() {
    syncService.forceSync();
  }
  
  Future getPendingCount() {
    return syncService.getPendingItemsCount();
  }

  @override
  void dispose() {
    syncService.syncData.removeListener(_onSyncStateChanged);
    super.dispose();
  }
}

📊 Monitoramento de Estado

Monitore o estado da sincronização em tempo real:

// No seu controller ou widget
void _setupSyncMonitoring() {
  final syncService = SyncConfigurator.syncService;
  
  syncService.syncData.addListener(() {
    final syncData = syncService.syncData.value;
    
    switch (syncData.status) {
      case AppSyncStatus.idle:
        print('Sincronização inativa');
        break;
      case AppSyncStatus.syncing:
        print('Sincronizando... ${syncData.pendingItems} itens pendentes');
        break;
      case AppSyncStatus.success:
        print('Sincronização concluída com sucesso!');
        _reloadData();
        break;
      case AppSyncStatus.error:
        print('Erro na sincronização: ${syncData.message}');
        break;
      case AppSyncStatus.offline:
        print('Modo offline - dados serão sincronizados quando conectar');
        break;
    }
  });
  
  // Verificar status online/offline
  final isOnline = await syncService.isOnline();
  final isSyncing = await syncService.isSyncing();
  final pendingCount = await syncService.getPendingItemsCount();
  
  print('Online: $isOnline, Sincronizando: $isSyncing, Pendentes: $pendingCount');
}

📊 Sistema de Envio de Erros para o Back-end

🎯 Objetivo do Sistema de Erros

O Syncly possui um sistema automático de coleta e envio de erros para o back-end, permitindo monitoramento em tempo real da saúde da sincronização e identificação proativa de problemas nos dispositivos dos usuários.

⏰ Quando os Erros São Enviados

🚨 Erros Críticos (Automático)

Enviados automaticamente quando o sistema atinge 3 falhas consecutivas de sincronização (SyncConstants.maxRetryAttempts).

  • • Falhas de conectividade persistentes
  • • Erros de autenticação recorrentes
  • • Problemas de servidor (5xx)
  • • Timeouts repetidos

⚠️ Erros Gerais (Programado)

Enviados em lotes através do SyncErrorReporter.scheduleErrorReporting()

  • • Erros de validação de dados
  • • Falhas de upload de arquivos
  • • Problemas de parsing JSON
  • • Erros de configuração

🔄 Como Funciona o Envio de Erros

1. Coleta e Armazenamento Local

SyncErrorManager.logError() → Salva erro localmente com metadata

↓

IStorageProvider → Persiste no storage local do dispositivo

2. Verificação de Condições

SyncErrorReporter.sendPendingErrors() → Verifica autenticação

↓

isAuthenticated() → Só envia se usuário autenticado

3. Envio em Lotes

_createBatches() → Agrupa erros (padrão: 10 por lote)

↓

_sendErrorBatch() → Envia via HTTP POST

↓

markErrorAsSent() → Marca como enviado se sucesso

⚙️ Configuração do Endpoint de Erros

🔍 Monitoramento e Retry

🔄 Sistema de Retry

  • • 3 tentativas por lote de erros
  • • 5 segundos de delay entre tentativas
  • • Erros não enviados ficam como pending
  • • Próxima execução tenta novamente

📊 Status de Envio

  • • SyncErrorReportResult retorna estatísticas
  • • sentCount e failedCount
  • • sentErrorIds e failedErrorIds
  • • Logs detalhados para debugging

🔧 Solução de Problemas

❌ Dados não estão sendo sincronizados

Possíveis Causas:

  • • Esqueceu de chamar logCreate, logUpdate ou logDelete
  • • SyncConfigurator não foi inicializado no main.dart
  • • Problemas de conectividade ou configuração de rede
  • • Erro na implementação do toJson() do modelo
  • • Configuração incorreta no SyncConfig

✅ Soluções:

  • • Verifique se todos os métodos CRUD chamam os respectivos métodos de log
  • • Confirme que SyncConfigurator.initialize() foi chamado no main.dart
  • • Teste a conectividade e verifique as configurações de URL no SyncConfig
  • • Valide se o toJson() retorna dados válidos
  • • Verifique se o token de autenticação está sendo retornado corretamente

❌ Erro de inicialização do sistema

Erros Comuns:

SyncConfigurator not initialized
GetIt: Object/factory with type AppSyncService is not registered

✅ Solução:

Verifique se o SyncConfigurator foi inicializado no main.dart:

void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  
  // Inicializar o sistema de sincronização
  await SyncConfigurator.initialize(AppSyncConfig());
  
  runApp(MyApp());
}

❌ Estado de sincronização não atualiza

Possíveis Causas:

  • • Listener não foi adicionado ao syncData
  • • Listener não foi removido no dispose()
  • • Controller não está sendo reconstruído
  • • SyncConfigurator não foi inicializado

✅ Solução:

// Acessar o serviço através do configurador
final syncService = SyncConfigurator.syncService;

// Adicionar listener
syncService.syncData.addListener(_onSyncStateChanged);

// Remover no dispose
@override
void dispose() {
  SyncConfigurator.syncService.syncData.removeListener(_onSyncStateChanged);
  super.dispose();
}
